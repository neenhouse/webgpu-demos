# Scene Pipeline Specification v1.0

## 1. Introduction

This document defines a declarative scene description format for procedural 3D content generation. A scene is described in a single YAML file that fully specifies geometry, materials, lighting, animation, and instancing -- everything a renderer needs to produce a complete 3D environment.

### 1.1 Goals

1. **Human-readable, machine-writable** -- YAML scenes can be authored by hand or generated by AI/scripting tools.
2. **Engine-agnostic** -- the format describes *what* to render, not *how*. Any 3D engine can implement a compatible renderer.
3. **Prompt-first** -- every object and material starts from a natural-language prompt. Structured parameters refine the prompt, never replace it.
4. **Composable** -- objects nest, materials inherit, prefabs deduplicate, instances multiply.
5. **Gracefully degradable** -- every optional feature has a defined fallback so partial implementations still produce visible output.

### 1.2 Design Principles

- **Prompts are primary, parameters are refinements.** An object's `prompt` field is the source of truth. Structured fields like `params`, `material.pbr`, and `generator` exist to guide and constrain interpretation, not to replace the prompt.
- **Convention over configuration.** Sensible defaults for every optional field. A minimal scene with just `meta`, one `object` with `id` and `prompt` is valid.
- **Fail visibly.** When generation fails, render a visible error marker rather than silently dropping content.
- **Inheritance is explicit.** Children do not silently inherit parent properties. Inheritance must be declared via `inherit: parent`.

### 1.3 Scope

This spec covers:
- The YAML scene file format (syntax and semantics)
- Object composition and transform hierarchy
- Material definition, resolution order, and preset catalog
- Texture specification
- Animation property targeting
- Geometry generator interface and selection
- Prefab registration and GPU instancing
- Level of detail (LOD) convention
- Error handling and fallback behavior

This spec does NOT cover:
- Physics simulation
- Audio
- User interaction / input handling
- Networking or multiplayer
- Specific rendering API calls

### 1.4 Terminology

| Term | Definition |
|------|-----------|
| Scene | A complete 3D environment described by a single YAML file |
| Object | A node in the scene graph -- may represent geometry, a group, or a prefab instance |
| Generator | A module that converts an object description into renderable geometry |
| Material | Surface appearance properties (color, roughness, metalness, shaders) |
| Preset | A named, reusable material configuration |
| Prefab | A registered object template that can be instanced multiple times |
| PBR | Physically Based Rendering -- the standard material model |
| TSL | Three Shading Language -- a node-based shader language (used in the reference implementation) |
| SDF | Signed Distance Field -- a mathematical shape representation rendered via raymarching |
| CSG | Constructive Solid Geometry -- shapes built from boolean operations on primitives |
| LOD | Level of Detail -- progressively simplified geometry for distant objects |
| Vec3 | A 3-component vector represented as `[x, y, z]` |

## 2. Scene File Format

Scene files use YAML syntax and follow the naming convention `<name>.scene.yaml`.

### 2.1 Top-Level Structure

```yaml
version: "1.0"              # REQUIRED. Spec version for forward compatibility.

meta:                        # REQUIRED. Scene metadata.
  name: string               #   REQUIRED. Display name.
  technique: string          #   REQUIRED. Fidelity label (e.g., "parametric + CSG").
  description: string        #   REQUIRED. One-line summary.
  author: string             #   Optional. Who or what generated this scene.

camera:                      # Optional. Defaults provided below.
  position: [x, y, z]       #   Default: [0, 5, 10]
  target: [x, y, z]         #   Default: [0, 0, 0]
  fov: number                #   Degrees. Default: 60
  near: number               #   Near clip plane. Default: 0.1
  far: number                #   Far clip plane. Default: 1000

environment:                 # Optional. Defaults provided below.
  description: string        #   Optional. Natural language description for AI generation.
  background: string         #   Hex color or CSS color string. Default: "#000000"
  fog: FogDef                #   Optional. See section 2.2.
  ambient: AmbientDef        #   Optional. See section 2.3.
  lights: Light[]            #   Optional. Default: [] (no lights). See section 2.4.

objects: Object[]            # REQUIRED. At least one object. See section 3.

prefabs:                     # Optional. Named prefab templates.
  <prefab-id>: Object        #   Each value is a full Object definition.
```

### 2.2 FogDef

```yaml
fog:
  type: linear | exponential   # REQUIRED.
  color: string                # REQUIRED. Hex color.
  near: number                 # Linear fog only. Distance where fog begins.
  far: number                  # Linear fog only. Distance where fog is fully opaque.
  density: number              # Exponential fog only. Density factor.
```

**Constraints:**
- When `type` is `linear`, `near` and `far` should be provided.
- When `type` is `exponential`, `density` should be provided.
- `near` and `far` are ignored for exponential fog; `density` is ignored for linear fog.

### 2.3 AmbientDef

```yaml
ambient:
  color: string                # Hex color. Default: "#ffffff"
  intensity: number            # Brightness multiplier. Default: 0.5
```

### 2.4 Light

```yaml
light:
  type: directional | point | spot | hemisphere   # REQUIRED.
  position: [x, y, z]         # Optional. World position.
  target: [x, y, z]           # Optional. Look-at target (directional and spot only).
  color: string                # Optional. Hex color.
  intensity: number            # Optional. Brightness multiplier.
  distance: number             # Optional. Falloff distance (point and spot only).
  angle: number                # Optional. Cone angle in degrees (spot only).
  castShadow: boolean          # Default: false.
```

**Type-specific behavior:**
- `directional`: parallel light from `position` toward `target`. No falloff.
- `point`: omnidirectional light at `position`. Falls off over `distance`.
- `spot`: cone of light from `position` toward `target` with `angle` aperture.
- `hemisphere`: sky/ground gradient light. `color` is the sky color.

## 3. Object Model

### 3.1 Object Schema

Every object in the scene follows this schema:

| Field | Type | Default | Required | Description |
|-------|------|---------|----------|-------------|
| `id` | string | -- | Yes | Unique identifier within the scene. Used for prefab references and material inheritance. |
| `prompt` | string | -- | Yes | Natural-language description of the object. This is the primary input for geometry generation. |
| `style` | enum | -- | No | Visual style hint. One of: `realistic`, `stylized`, `cel-shaded`, `low-poly`, `voxel`, `wireframe`. |
| `generator` | string | -- | No | Generator hint. Examples: `csg`, `parametric/terrain`, `parametric/vehicle`, `sdf`, `codegen`, `tripo`. See section 7. |
| `params` | map | -- | No | Generator-specific parameters. Free-form key-value pairs. |
| `transform` | Transform | see below | No | Position, rotation, and scale. See section 3.2. |
| `material` | MaterialDef | -- | No | Surface material definition. See section 4. |
| `textures` | TextureDef | -- | No | Texture configuration. See section 5. |
| `children` | Object[] | -- | No | Child objects. Transforms are parent-local. See section 3.3. |
| `animation` | Animation[] | -- | No | Animations applied to this object. See section 6. |
| `register_prefab` | boolean | -- | No | If `true`, register this object as a reusable prefab after generation. |
| `prefab_ref` | string | -- | No | Reference a prefab by its `id`. Geometry and material are cloned from the prefab. |
| `instances` | Transform[] | -- | No | GPU-instanced placements. Each entry is a Transform. See section 8. |
| `lod` | `"auto"` / `"none"` / LodDef | `"none"` | No | Level of detail. See section 9. |
| `collision` | enum | `"none"` | No | Collision shape hint (reserved for future physics). One of: `none`, `box`, `sphere`, `mesh`, `convex`. |
| `visible` | boolean | `true` | No | Whether the object is rendered. |
| `castShadow` | boolean | `true` | No | Whether the object casts shadows. |
| `receiveShadow` | boolean | `true` | No | Whether the object receives shadows. |

### 3.2 Transform

```yaml
transform:
  position: [x, y, z]         # World or parent-local coordinates. Default: [0, 0, 0]
  rotation: [x, y, z]         # Euler angles in degrees. Default: [0, 0, 0]
  scale: number | [x, y, z]   # Uniform scalar or per-axis. Default: 1
```

**Conventions:**
- Coordinate system: right-handed, Y-up. +X is right, +Y is up, +Z is toward the viewer.
- Rotation order: intrinsic XYZ (apply X, then Y, then Z).
- Scale `1` is equivalent to `[1, 1, 1]`.
- When `scale` is a single number, it applies uniformly to all axes.

### 3.3 Composition Rules (Children)

Objects may contain `children`, forming a scene graph hierarchy.

| Property | Inheritance Behavior |
|----------|---------------------|
| `transform` | Children transforms are **parent-local**. `position: [1, 0, 0]` means 1 unit from the parent origin, not from the world origin. |
| `style` | Children **inherit** the parent's `style` unless they specify their own. |
| `generator` | Children do **NOT** inherit `generator`. Each child must specify its own or rely on auto-selection. |
| `material` | Children do **NOT** inherit `material` unless explicitly declared via `material: { inherit: parent }`. |
| All others | Not inherited. Each child must declare its own values or accept defaults. |

**Example -- parent with children:**

```yaml
- id: car-wreck
  prompt: rusted abandoned sedan with flat tires
  style: realistic
  generator: parametric/vehicle
  transform:
    position: [3, 0, -2]
    rotation: [0, 35, 0]
  material:
    preset: rusted-metal
  children:
    - id: car-tire-fl
      prompt: flat deflated tire
      generator: parametric/debris
      transform:
        position: [-0.7, 0.2, 0.9]    # relative to car-wreck
        scale: 0.3
      material:
        preset: rubber-worn            # own material (not inherited)
    - id: car-hood
      prompt: dented hood panel
      generator: csg
      transform:
        position: [0, 0.5, 0.8]
      material:
        inherit: parent                # explicitly inherits car-wreck's material
        overrides:
          roughness: 0.95              # more worn than parent
```

## 4. Material System

### 4.1 MaterialDef Schema

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `prompt` | string | -- | Natural-language material description. |
| `preset` | string | -- | Named preset from the material library. See section 11. |
| `pbr` | PbrDef | -- | Structured PBR property overrides. See section 4.2. |
| `shader` | string | -- | Inline shader code block for custom materials. |
| `inherit` | string | -- | `"parent"` to inherit from parent object, or an object `id` string. |
| `overrides` | map | -- | PBR property overrides applied last (for inheritance scenarios). |
| `side` | enum | `"front"` | Face rendering. One of: `front`, `back`, `double`. |
| `transparent` | boolean | -- | Enable transparency. |
| `blending` | enum | `"normal"` | Blend mode. One of: `normal`, `additive`. |
| `wireframe` | boolean | -- | Render as wireframe. |
| `flatShading` | boolean | -- | Use flat (faceted) shading instead of smooth. |

### 4.2 PbrDef

| Field | Type | Range | Default | Description |
|-------|------|-------|---------|-------------|
| `color` | string | -- | -- | Base color. Hex (`"#8B4513"`) or CSS color name. |
| `roughness` | number | 0.0 - 1.0 | -- | Surface roughness. 0 = mirror, 1 = fully diffuse. |
| `metalness` | number | 0.0 - 1.0 | -- | Metallic factor. 0 = dielectric, 1 = metal. |
| `opacity` | number | 0.0 - 1.0 | -- | Opacity. 0 = invisible, 1 = fully opaque. Setting this implies `transparent: true`. |
| `emissive` | string | -- | -- | Emissive color. Hex string. |
| `emissive_intensity` | number | >= 0 | -- | Emissive brightness multiplier. |

**Constraints:** Values outside `[0, 1]` for `roughness`, `metalness`, and `opacity` must be clamped to the valid range by the renderer.

### 4.3 Material Resolution Order

Materials resolve in strict priority order. Each step builds on or overrides the previous:

| Priority | Source | Behavior |
|----------|--------|----------|
| 1 (lowest) | `inherit` | Clone the material from the referenced parent or named object. |
| 2 | `preset` | Load a named preset as the base material. |
| 3 | `pbr` | Override specific PBR properties on the base. |
| 4 | `prompt` | AI/heuristic interprets the prompt to fill any remaining unset PBR values. |
| 5 | `shader` | If present, **replaces the entire shader graph**. PBR scalar values (`roughness`, `metalness`) still apply alongside the shader. |
| 6 (highest) | `overrides` | Final property overrides, applied last. Intended for inheritance scenarios. |

**Key rules:**
- If no `inherit`, `preset`, `pbr`, `prompt`, or `shader` is specified, the renderer creates a default grey PBR material.
- `shader` is **atomic** -- you cannot inherit a shader and override individual lines. To modify a shader, provide a complete replacement.
- `overrides` uses the same field names as `pbr` (`color`, `roughness`, `metalness`, `opacity`, `emissive`, `emissive_intensity`).

**Example -- layered material resolution:**

```yaml
material:
  inherit: parent              # Step 1: clone parent material
  preset: rusted-metal         # Step 2: override with rusted-metal preset
  pbr:                         # Step 3: override specific PBR values
    roughness: 0.95
    color: "#7B3F00"
  overrides:                   # Step 6: final adjustments
    emissive_intensity: 0.2
```

### 4.4 Shader Extension Point

The `shader` field accepts inline shader code as a YAML block scalar. The shader code operates on a material object (`mat`) and can assign node-based properties:

```yaml
material:
  shader: |
    mat.colorNode = mix(
      color(0x003300),
      color(0x00ff88),
      positionLocal.y.mul(0.5).add(0.5)
    );
    mat.emissiveNode = color(0x00ffcc).mul(
      oscSine(time.add(positionLocal.y.mul(3.0))).mul(0.5).add(0.5)
    );
```

**Available shader targets** (node graph assignments):
- `mat.colorNode` -- base surface color
- `mat.emissiveNode` -- emissive color and intensity
- `mat.roughnessNode` -- roughness (scalar node)
- `mat.metalnessNode` -- metalness (scalar node)
- `mat.opacityNode` -- opacity (scalar node)
- `mat.normalNode` -- normal map perturbation
- `mat.positionNode` -- vertex displacement

Implementations should provide standard shader utilities:
- `color(hex)` -- construct a color
- `float(n)` -- construct a scalar
- `mix(a, b, factor)` -- linear interpolation
- `positionLocal` -- object-space vertex position
- `positionWorld` -- world-space vertex position
- `normalLocal` -- object-space normal
- `normalWorld` -- world-space normal
- `time` -- elapsed time in seconds
- `oscSine(value)` -- sine oscillator (0-1 range)
- `hash(value)` -- pseudo-random hash noise

## 5. Texture System

### 5.1 TextureDef Schema

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `prompt` | string | -- | Natural-language description of desired textures. |
| `maps` | string[] | -- | Which PBR maps to generate. See allowed values below. |
| `resolution` | number | `1024` | Texture resolution in pixels (width and height). |
| `tiling` | [number, number] | `[1, 1]` | UV tiling factor `[u, v]`. |
| `source` | enum | `"procedural"` | Texture generation method. One of: `procedural`, `ai-generated`, `file`. |
| `paths` | map | -- | File paths for each map when `source` is `"file"`. Keys are map names, values are file paths. |

**Allowed map types:**
- `albedo` -- base color texture
- `normal` -- normal map for surface detail
- `roughness` -- per-pixel roughness
- `metalness` -- per-pixel metalness
- `ao` -- ambient occlusion
- `emission` -- emissive areas
- `displacement` -- height displacement

**Example:**

```yaml
textures:
  prompt: weathered concrete with cracks and moss
  maps: [albedo, normal, roughness, ao]
  resolution: 2048
  tiling: [2, 2]
  source: procedural
```

**File-based textures:**

```yaml
textures:
  source: file
  tiling: [1, 1]
  paths:
    albedo: textures/brick_albedo.png
    normal: textures/brick_normal.png
    roughness: textures/brick_roughness.png
```

## 6. Animation System

### 6.1 Animation Schema

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `property` | string | -- | **Required.** Dot-path to the animated property. See section 6.2. |
| `type` | enum | -- | **Required.** Evaluator type. One of: `sine`, `bounce`, `rotate`, `sway`, `pulse`, `custom`. |
| `speed` | number | `1` | Cycles per second. |
| `amplitude` | number | `1` | Magnitude of the effect. |
| `range` | [number, number] | -- | Optional. Clamp output to `[min, max]`. |
| `delay` | number | `0` | Seconds before the animation starts. |
| `loop` | boolean | `true` | Whether the animation repeats. |

### 6.2 Property Targets

Valid `property` values use dot-notation to address nested object fields:

| Property Path | Effect |
|---------------|--------|
| `transform.position.x` | Translate along X axis |
| `transform.position.y` | Translate along Y axis |
| `transform.position.z` | Translate along Z axis |
| `transform.rotation.x` | Rotate around X axis (degrees) |
| `transform.rotation.y` | Rotate around Y axis (degrees) |
| `transform.rotation.z` | Rotate around Z axis (degrees) |
| `transform.scale` | Uniform scale |
| `material.pbr.emissive_intensity` | Emissive brightness pulse |
| `material.pbr.opacity` | Fade in/out |
| `material.pbr.roughness` | Surface variation over time |
| `material.pbr.metalness` | Metalness variation over time |
| `visibility` | Show/hide toggle |

**Unknown property paths must be silently ignored** by the renderer to allow forward compatibility.

### 6.3 Evaluator Types

| Type | Behavior |
|------|----------|
| `sine` | Smooth sinusoidal oscillation. Output = `sin(time * speed * 2pi) * amplitude` |
| `bounce` | Absolute sine for bouncing effect. Output = `|sin(time * speed * 2pi)| * amplitude` |
| `rotate` | Continuous rotation. Adds `speed * amplitude * deltaTime` to the current value each frame. |
| `sway` | Gentle back-and-forth. Similar to `sine` but with damping for natural plant-like motion. |
| `pulse` | Pulsing glow effect. Sine wave biased toward the upper range. |
| `custom` | Implementation-defined evaluator. The `params` field on the parent object may contain custom evaluator configuration. |

When `range` is specified, the evaluator output is clamped to `[min, max]` after computation.

**Example:**

```yaml
animation:
  - property: transform.rotation.y
    type: rotate
    speed: 0.1
    amplitude: 360
  - property: material.pbr.emissive_intensity
    type: pulse
    speed: 0.5
    amplitude: 1.5
    range: [0.3, 1.8]
```

## 7. Generator Interface

### 7.1 Overview

Geometry generators convert an object's `prompt` (plus optional `generator` hint and `params`) into renderable geometry. The `generator` field is a **hint** -- the renderer may use it directly, or run an auto-selection process based on confidence scoring.

### 7.2 Generator Interface (Abstract)

Every generator must implement:

```
Generator:
  name: string
  canHandle(object: Object) -> confidence: number   # 0.0 to 1.0
  generate(object: Object) -> GeneratorResult
```

**GeneratorResult:**

| Field | Type | Description |
|-------|------|-------------|
| `geometry` | BufferGeometry | The generated mesh (or bounding box for SDF). |
| `material` | Material or null | If provided, the renderer skips the material pipeline for this object. |
| `isSdf` | boolean | If `true`, the geometry is a bounding volume and the material contains a raymarching shader. |
| `metadata.vertexCount` | number | Vertex count of the generated mesh. |
| `metadata.faceCount` | number | Face/triangle count. |
| `metadata.generator` | string | Name of the generator that produced this result. |
| `metadata.prompt` | string | The prompt that was used. |
| `metadata.generationTime` | number | Generation time in milliseconds. |

### 7.3 Confidence-Based Selection

When no `generator` hint is specified (or the hint is ambiguous), the renderer queries all available generators via `canHandle()` and selects the one with the highest confidence score.

```
For each registered generator:
  score = generator.canHandle(object)
Select generator with highest score where score > 0
If all scores are 0: use fallback (error marker)
```

### 7.4 The Five Generator Tiers

| Tier | Name | Generator Hint | Best For | Output |
|------|------|---------------|----------|--------|
| 1 | CSG | `csg` | Mechanical objects, architecture, barriers, containers. Simple shapes from boolean operations on primitives. | Mesh geometry |
| 2 | Parametric | `parametric/<category>` | Category-specific objects with rich parameter control. | Mesh geometry with UVs |
| 3 | SDF Raymarching | `sdf` | Organic, alien, abstract, sci-fi shapes. Infinite variety from math. | Bounding box + raymarching shader |
| 4 | AI Code Generation | `codegen` | Unique manufactured objects that do not fit standard categories (Victorian lamp, ornate gate). | Mesh geometry |
| 5 | External API Import | `tripo` (or other API name) | Photorealistic objects when an external generation API is available. | Imported mesh |

#### 7.4.1 Tier 1: CSG Generator

Constructive Solid Geometry -- boolean operations on geometric primitives.

**Primitives:** `box`, `sphere`, `cylinder`, `cone`, `torus`

**Operations:** `union`, `subtract`, `intersect`

**Params example:**

```yaml
generator: csg
params:
  operations:
    - union:
        - box: [2, 0.8, 0.4]
        - cylinder: [0.1, 1]
          position: [0.5, 0, 0]
    - subtract:
        - sphere: [0.3]
          position: [0, 0.4, 0.2]
```

Each primitive in the operations list accepts its defining dimensions as an array and an optional `position` offset.

#### 7.4.2 Tier 2: Parametric Generators

Category-specific generators with rich parameter control. The `generator` hint uses the format `parametric/<category>`.

**Standard categories:**

| Category | Generator Hint | Typical Objects |
|----------|---------------|-----------------|
| Vehicle | `parametric/vehicle` | Cars, trucks, motorcycles |
| Character | `parametric/character` | Humanoids, creatures, robots |
| Vegetation | `parametric/vegetation` | Trees, bushes, grass, flowers |
| Terrain | `parametric/terrain` | Ground planes, hills, cliffs |
| Rock | `parametric/rock` | Rocks, boulders, rubble |
| Building | `parametric/building` | Structures, walls, ruins |
| Furniture | `parametric/furniture` | Tables, chairs, shelves |
| Debris | `parametric/debris` | Tires, barrels, crates, pipes |
| Weapon | `parametric/weapon` | Swords, guns, shields |
| Organic | `parametric/organic` | Mushrooms, corals, alien life |

Each parametric generator:
- Accepts a `prompt` plus optional `params` specific to its category
- Returns geometry with proper UV coordinates suitable for texturing
- Respects the `style` field (affects polygon count and edge treatment)
- Produces prefab-ready meshes suitable for instancing

#### 7.4.3 Tier 3: SDF Raymarching

Signed Distance Field shapes rendered via raymarching in the fragment shader.

**Key characteristic:** The SDF generator returns **both geometry and material** (`isSdf: true`). The geometry is a bounding box mesh sized to contain the SDF shape. The material contains a raymarching fragment shader. The renderer **skips the material pipeline** for SDF objects.

**Built-in SDF primitives:** `sphere`, `box`, `torus`, `cylinder`, `cone`, `capsule`

**Built-in SDF operations:** `union`, `smooth_union`, `subtract`, `intersect`, `repeat`, `twist`, `bend`

Color and material for SDF objects are embedded in the SDF shader (e.g., color based on distance field value, position, or normal), defined in the object's `material.shader` field.

#### 7.4.4 Tier 4: AI Code Generation

For unique objects that do not fit parametric categories. A code generation system writes a geometry factory function from the prompt.

```yaml
generator: codegen
params:
  source: "generated/junkyard/victorian-lamp.ts"    # path to generated factory
```

#### 7.4.5 Tier 5: External API Import

Import meshes from external 3D generation APIs.

```yaml
generator: tripo
params:
  api_prompt: "detailed police car, photorealistic"
  cache_key: "police-car-v1"
```

- Results are cached locally after first generation.
- Falls back to Tier 4 if the API is unavailable.

### 7.5 Generator Selection Heuristic

When a generator writes a scene (manually or via AI), the recommended selection heuristic is:

1. Does the prompt match a parametric category? -> Tier 2
2. Is the object organic, abstract, alien, or sci-fi? -> Tier 3 (SDF)
3. Is the object mechanical or composed of simple shapes? -> Tier 1 (CSG)
4. Is the object a unique manufactured item? -> Tier 4 (AI code gen)
5. Is photorealism required and an API available? -> Tier 5 (external import)

## 8. Prefab and Instancing

### 8.1 Prefab Registration

An object with `register_prefab: true` is registered as a reusable prefab after its geometry and material are generated. Other objects can reference it via `prefab_ref`.

```yaml
- id: barrel
  prompt: rusty oil barrel
  generator: parametric/debris
  material:
    preset: rusted-metal
  register_prefab: true         # Makes this available as a prefab

- id: barrel-copy
  prompt: another barrel
  prefab_ref: barrel             # Clones geometry and material from "barrel"
  transform:
    position: [5, 0, 3]
```

**Prefab sources:**

1. **Inline** -- `register_prefab: true` on an object in the `objects` list.
2. **Prefab section** -- defined in the top-level `prefabs:` map.
3. **Cross-scene** -- implementations may support loading prefabs from external files (implementation-defined).

### 8.2 GPU Instancing

When an object has an `instances` array, the renderer:

1. Generates the mesh **once** (from the object definition or prefab).
2. Creates an instanced mesh with N instances (where N = length of `instances`).
3. Sets per-instance transform matrices from the `instances` array.

```yaml
- id: rock-scatter
  prompt: weathered rubble
  generator: parametric/rock
  material:
    pbr:
      color: "#666660"
      roughness: 0.9
  register_prefab: true
  instances:
    - position: [2, 0, 7]
      rotation: [0, 0, 0]
      scale: 0.6
    - position: [-5, 0, 6]
      rotation: [0, 45, 0]
      scale: 0.4
    - position: [6, 0, -7]
      rotation: [0, 120, 0]
      scale: 0.8
```

**Instancing rules:**
- All instances share the same geometry and material.
- Per-instance color or material variation is implementation-defined (not covered by this spec).
- The object's own `transform` is ignored when `instances` is present -- only the instance transforms are used.

## 9. LOD Convention

### 9.1 LOD Modes

| Mode | Behavior |
|------|----------|
| `"none"` | No LOD. Always render full detail. This is the default. |
| `"auto"` | Renderer generates 3 detail levels automatically (see below). |
| LodDef object | Custom detail levels defined by the scene author. |

### 9.2 Auto LOD

When `lod: auto`, the renderer generates three levels:

| Level | Camera Distance | Detail Factor | Description |
|-------|----------------|---------------|-------------|
| High | 0 - 30 units | 1.0 | Original generated geometry. |
| Medium | 30 - 80 units | 0.5 | Simplified to ~50% of original faces. |
| Low | 80+ units | 0.2 | Simplified to ~20% of original faces. |

### 9.3 Custom LOD

```yaml
lod:
  levels:
    - distance: 0              # Camera distance threshold
      detail: 1.0              # Proportion of original detail (0.0 - 1.0)
    - distance: 30
      detail: 0.5
    - distance: 80
      detail: 0.2
```

**LodDef:**

| Field | Type | Description |
|-------|------|-------------|
| `levels` | LodLevel[] | **Required.** At least one level. |
| `levels[].distance` | number | **Required.** Camera distance at which this level activates. |
| `levels[].detail` | number | **Required.** 0.0 to 1.0. Proportion of original geometry detail. |

### 9.4 LOD Interactions

- **LOD + Instancing:** LOD does not apply to instanced meshes in most renderers. All instances use the same detail level. An implementation may partition instances by camera distance as an optimization, but this is not required.
- **LOD + SDF:** SDF objects do not need mesh LOD. Implementations may reduce raymarching iteration count or simplify the SDF function based on camera distance.

## 10. Error Handling

Every failure mode has a defined fallback behavior to ensure scenes degrade gracefully.

| Failure | Fallback Behavior |
|---------|-------------------|
| **YAML parse error** | Reject the entire scene. Display a descriptive error message with line/column information. |
| **Missing required field** (`id`, `prompt`) | Reject the scene with a field-level validation error identifying the missing field and its location. |
| **Generator failure** (crash, degenerate geometry) | Render a **bright magenta wireframe cube** (`#FF00FF`) at the object's position and scale as a visible error marker. Log the error. |
| **Unknown generator hint** | Fall through to confidence-based auto-selection. If no generator can handle the object, render the error marker. |
| **Unknown material preset** | Fall back to a default grey PBR material (roughness: 0.5, metalness: 0.0, color: grey). Log a warning with the unknown preset name. |
| **Unknown prefab reference** | Skip the object entirely. Log an error with the unresolved `prefab_ref` value. |
| **Invalid PBR values** (out of range) | Clamp to the valid range. `roughness: 1.5` becomes `1.0`; `opacity: -0.3` becomes `0.0`. |
| **Missing parent for `inherit: parent`** | Create a default grey material. Log a warning. |
| **Missing referenced material for `inherit: <id>`** | Create a default grey material. Log a warning with the unresolved ID. |
| **Shader compilation error** | Fall back to the PBR material (skip the shader step). Log the compilation error. |
| **Texture generation failure** | Proceed without textures. The material PBR properties provide the baseline appearance. |
| **Animation on unknown property** | Silently ignore the animation. Do not warn (forward compatibility). |

**Error marker specification:**
- Geometry: unit cube (1x1x1), rendered as wireframe
- Color: `#FF00FF` (bright magenta)
- Position and scale: match the failed object's transform
- The marker must be obviously wrong to scene authors -- it is intentionally ugly

## 13. Appendix: Reference Implementation

The reference implementation of this specification is the **WebGPU Demos** project, which uses:

- **Renderer:** Three.js WebGPURenderer (with WebGLRenderer fallback)
- **Schema validation:** Zod runtime validation from `src/pipeline/spec/schema.ts`
- **Material system:** MeshStandardNodeMaterial with TSL (Three Shading Language) node graphs
- **Generators:** CSG via three-bvh-csg, parametric BufferGeometry builders, SDF via TSL raymarching
- **Instancing:** Three.js InstancedMesh
- **LOD:** Three.js LOD object

Source code and scene examples are available at the project repository. The reference implementation covers Tiers 1-4 of the generator system, all 14 material presets, the full material resolution chain, animation evaluation, prefab registration, GPU instancing, and LOD.

### Conformance

An implementation is conformant with this spec if it:

1. Parses valid scene YAML files without error
2. Rejects invalid scenes with descriptive error messages
3. Renders all objects with visible geometry (or error markers on failure)
4. Resolves materials in the documented priority order
5. Supports at least one generator tier
6. Handles all error cases per section 10
7. Supports `instances` for GPU instancing
8. Supports `children` with parent-local transforms

Partial implementations are expected and encouraged. A renderer that only supports Tier 2 (parametric) generators is useful. A renderer that skips LOD is functional. The error handling rules ensure graceful degradation.
