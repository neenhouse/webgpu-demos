## Question

What best practices from Three.js/WebGPU/R3F communities can improve our scene quality, performance, and pipeline? What anti-patterns exist in our current 146 demos?

## Context

The project has 146 WebGPU demos across 12 themed batches. Most were batch-generated by AI subagents, which followed the learnings.md and batch-playbook.md but may have introduced performance anti-patterns not covered by those docs. The quality audit script checks visual rules (emissive limits, DoubleSide, line counts) but not runtime performance patterns.

## Sources Consulted

1. Three.js official docs (WebGPU renderer, BatchedMesh, InstancedMesh)
2. React Three Fiber docs (performance scaling, pitfalls, re-renders)
3. WebGPU Three.js TSL Skill (dgreenheck/webgpu-claude-skill)
4. R3F ESLint plugin (no-new-in-loop rule)
5. Static analysis of all 146 demo source files

## Findings

### Anti-Pattern 1: Object Allocation in useFrame (CRITICAL — 40 demos)

Creating `new THREE.Object3D()`, `new THREE.Vector3()`, `new THREE.Matrix4()` etc. inside `useFrame` callbacks. These allocate on every frame (60x/sec), pressuring the GC and causing frame stutters.

**Fix**: Hoist to `useRef` or `useMemo` outside the frame loop. Use a single `dummy` Object3D per component.

**Example bad**:
```tsx
useFrame(() => {
  const dummy = new THREE.Object3D(); // BAD: allocates every frame
  for (let i = 0; i < COUNT; i++) {
    dummy.position.set(x, y, z);
    dummy.updateMatrix();
    mesh.setMatrixAt(i, dummy.matrix);
  }
});
```

**Example good**:
```tsx
const dummy = useMemo(() => new THREE.Object3D(), []);
useFrame(() => {
  for (let i = 0; i < COUNT; i++) {
    dummy.position.set(x, y, z);
    dummy.updateMatrix();
    mesh.setMatrixAt(i, dummy.matrix);
  }
});
```

### Anti-Pattern 2: Material Creation Outside useMemo (HIGH — 22 demos)

Materials created in helper functions or component body (not wrapped in useMemo) get recreated on every React render, wasting GPU resources.

**Fix**: Wrap all material creation in `useMemo(() => { ... }, [deps])`.

### Anti-Pattern 3: Missing frustumCulled={false} on InstancedMesh (MEDIUM — 70 demos)

Three.js performs per-instance frustum culling by default on InstancedMesh. For most of our demos (where all instances are visible), this is wasted CPU work. Disabling it avoids the overhead.

**Fix**: Add `frustumCulled={false}` to all `<instancedMesh>` elements.

### Anti-Pattern 4: Duplicate Geometry Creation (MEDIUM — 34 demos)

Same geometry type (e.g., `<sphereGeometry args={[1, 6, 6]} />`) created multiple times within the same component. Each creates a separate GPU buffer.

**Fix**: Create geometry once in `useMemo` and pass via `geometry` prop.

### Positive Finding: No Uninstanced Large Mesh Sets

Zero demos create 50+ individual `<mesh>` elements — all use InstancedMesh correctly for bulk rendering. The batch playbook's instancing guidance worked.

## Recommendation

**Fix all 4 anti-patterns.** Priority order:

1. **Object allocation in useFrame** (40 demos) — biggest perf impact
2. **Material creation outside useMemo** (22 demos) — GPU resource waste
3. **Missing frustumCulled={false}** (70 demos) — easy fix, broad impact
4. **Duplicate geometry** (34 demos) — lower priority, but good hygiene

**Update playbook** with new rules:
- NEVER create `new THREE.*` inside useFrame — use useRef/useMemo
- Materials MUST be wrapped in useMemo
- InstancedMesh MUST have frustumCulled={false}
- Share geometry via useMemo when same type appears 3+ times

**Update quality-audit.mjs** to detect these patterns.
